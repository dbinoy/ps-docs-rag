# 2-Step Security Brief

Overview

This document set describes a step-up authentication layer protecting sensitive applications (e.g., User Management) and functionality within Central 1's systems. It's a mandatory second factor for access control.

Key Architectural Elements

Authentication Methods:

  • Hard tokens (physical devices) – ordered/managed by financial institutions via User Management
  • Soft tokens (RFC 6238 compliant authenticator apps) – user-installed on mobile devices

Data Flow:

  • Users must have a 2-step security token assigned to their Client Centre User ID
  • Authentication occurs at login or functionality access points
  • Token validation is required before granting access

Business Rules:

  • Security Officers manage hard token ordering and replacement
  • Soft token registration requires app installation + registration process
  • Token assignment is prerequisite for protected resource access
  • Troubleshooting assumes token-to-user matching validation

Design Constraints for Architects

1. Integration Points: User Management system handles token assignment and ordering workflows

2. Standards Compliance: Soft tokens must be RFC 6238 compliant (TOTP-based)

3. Operational Dependency: Hard token supply chain involves third parties

4. User Friction: Step-up authentication adds login friction—consider UX in enhanced designs

Critical Gap: Document lacks technical details on token generation, validation logic, session management, and API contracts needed for system integration.
