# CPS Admin Application - Architecture Brief

Overview

CPS Admin is Central 1's administrative interface for managing Client Centre user access and authentication. It serves as the control plane for identity and access management across high-security applications (e.g., Wires).

Key Architectural Components

User Management Flow:

  • CPS Admin Security Officers (minimum 2 per institution) create, modify, and delete Client Centre users
  • Users authenticate via 2-step security (step-up authentication)
  • Template-based user provisioning supported for bulk operations
  • Users self-register passwords through Client Centre login portal

Security Constraints:

  • Account lockout: 7 consecutive failed login attempts → 30-minute lock (persistent across sessions)
  • 2-step authentication mandatory for high-security application access
  • Security Officer role requires separate management policies
  • Access policies restrict application availability based on institution-level rules

Integration Points:

  • Client Centre (digital portal gateway)
  • User Management Application (parallel user administration)
  • Wires application (downstream consumer of access policies)

Design Considerations

  • State management: Track failed login attempts across distributed sessions
  • Audit trail: Security Officer actions (create/modify/delete) require logging
  • Policy engine: Institution-level access restriction enforcement
  • Redundancy: Dual Security Officer requirement demands high availability for user management

Critical gap: Document lacks technical details on user data schema, API contracts, and inter-system communication protocols.
