# Access and Administration – Section Overview

Purpose

This section defines the operational framework for user provisioning, authentication, and system access across Central 1's Payment Solutions platform. It covers identity and access management (IAM), multi-factor authentication architecture, file exchange integration points, and administrative governance policies that underpin all downstream payment processing systems.

Key Concepts

1. Client Centre Portal: Single entry point for all Central 1 applications; acts as the authentication gateway and session manager for downstream systems (Wires, User Management, File Exchange).

2. 2-Step Security (Step-Up Authentication): RFC 6238-compliant MFA layer protecting high-risk applications and functions. Uses either hardware tokens (ordered via ServiceNow) or soft tokens (Google Authenticator or compatible apps). One-time passwords required at login or for sensitive operations (e.g., wire approvals).

3. Role-Based Access Control (RBAC): Segregated permissions model with minimum two Security Officers per institution. Distinct roles include CPS Admin Security Officers, User Management Security Officers, and Client Centre Users, each with tiered privileges.

4. File and Report Exchange (SFTP + Web UI): Bidirectional file transfer mechanism supporting both direct SFTP connections and web-based uploads/downloads. Manages incoming AFT files, clearing files, and outgoing reconciliation/settlement reports.

5. Audit and Governance: Request/approval workflow for privilege changes, 30-minute account lockout after 7 failed login attempts, CSV-format reporting, and Business Continuity Program integration.

How It Works

1. User Onboarding: Institution appoints minimum two Security Officers via ServiceNow. Security Officers create Client Centre Users (individually or via templates) in CPS Admin or User Management.

2. Authentication: Users register passwords in Client Centre. High-security applications (Wires, User Management) enforce 2-step security; users must provision hardware or soft tokens beforehand.

3. Access Provisioning: Security Officers assign application permissions and 2-step tokens via User Management. Changes to user profiles or permissions trigger cross-officer approval workflows.

4. File Operations: Users access File Exchange via Client Centre UI (browser) or SFTP. Incoming files are uploaded to institution-specific folders; Central 1 generates outgoing files (clearing, settlement) posted for download.

Integration Points

  • ServiceNow: Hard token ordering and request management.
  • Client Centre: Primary identity provider; session manager for all protected applications.
  • User Management Application: Secondary access control interface; requires 2-step security.
  • File and Report Exchange: SFTP and HTTP protocols; separate access control policies.
  • Wires Application: Enforces step-up 2-step authentication for wire approvals.
  • Clearing/Settlement Batch Processes: Consume outgoing files from File Exchange.

Architect Notes

  • No centralized IdP abstraction: Client Centre authentication is tightly coupled to downstream applications; consider federation impact if integrating external identity providers.
  • Token provisioning latency: Hard tokens ship to institutions; soft tokens require manual app installation. Plan onboarding timelines accordingly.
  • File Exchange dual-protocol support: SFTP and web UI have different security postures; enforce consistent TLS/encryption standards and audit logging across both.
  • Compliance assumption: BCP references suggest regulatory reporting obligations; validate alignment with PIPEDA, provincial credit union oversight, and ACH/clearing house requirements.
  • Account lockout stateless: 30-minute lockout persists across sessions; no documented override for legitimate access recovery—escalation path to ServiceNow may be required.
