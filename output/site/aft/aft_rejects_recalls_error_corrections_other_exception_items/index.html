# AFT Exception Handling – Architect Brief

Overview

This document set governs the lifecycle management of Automated Funds Transfer (AFT) exceptions at Central 1, covering Pre-authorized Debits (PADs) and Direct Deposits. It defines handling protocols for rejects, recalls, error corrections, trace requests, and stop payments.

Key Architectural Concepts

Exception Categories:

  • Rejects: File or transaction-level formatting errors; transaction cannot post
  • Recalls: Pre-posting reversal of unposted transactions/files
  • Error Corrections: Post-posting adjustments to posted transactions
  • Trace Requests: Investigation mechanism for non-receipt at receiving institution
  • Stop Payments: Member-initiated PAD cancellation claims

Critical Data Flows

  • Inbound AFT files → validation → reject/recall queues vs. posting pipeline
  • Posted transactions → error correction audit trail with legal compliance
  • Outbound transactions → trace initiation if non-receipt reported by Due Date
  • Stop payment claims → reimbursement processing with member notification

Design Constraints

  • Regulatory compliance mandatory (statutes referenced in Part 3)
  • Clear distinction between pre-posting (recall) and post-posting (error correction) states
  • Audit trail requirements for all exception handling
  • Member notification/reimbursement workflows required
  • Due Date tracking critical for trace request triggering

Integration Points: File validation, posting engine, member communication systems, receiving institution coordination.
