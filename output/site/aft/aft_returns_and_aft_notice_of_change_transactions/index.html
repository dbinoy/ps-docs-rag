# AFT Returns and AFT Notice of Change Transactions – Architecture Brief

Overview

This document set governs the return and correction workflows for Automated Funds Transfer (AFT) transactions within Central 1's clearing network. It covers return scenarios, settlement mechanics, dispute handling, and change notifications.

Key Data Flows & Rules

AFT Returns:

  • Originating institutions can return transactions for: NSF, invalid/incorrect account numbers, or other defined rejection reasons
  • Settlement includes two models: offsetting return entries or standalone returns
  • Dishonoured transactions have defined recourse periods (especially for Business PADs)

AFT Notice of Change (NOC):

  • Non-financial transactions generated by receiving institutions to notify originators of account/recipient changes
  • Flows back through Originating Direct Clearers to originating institutions

Exception Handling:

  • Disputed PADs require Reimbursement Claim forms (Form 1701)
  • Batch Return Service enables electronic returns (BC jurisdiction)
  • Contingency procedures for file transmission failures

Critical Architecture Considerations

  • Dual settlement models require configurable processing logic
  • Multi-jurisdiction variations (BC-specific Batch Return Service)
  • Legal/compliance dependencies – design must accommodate evolving regulatory frameworks
  • Bidirectional messaging – returns and NOCs create reverse data flows
  • State management – recourse periods and dispute windows require temporal tracking
