# AFT Section Overview

Purpose

The AFT section establishes the complete operational framework for Automated Funds Transfer processing within Central 1's payment network. It covers origination, settlement, exception handling, and reporting for both Pre-authorized Debits (PADs) and Direct Deposits—the two primary AFT transaction types. This section is foundational to understanding how electronic credit and debit transactions flow through the payment infrastructure.

Key Concepts

1. AFT Transaction Types: PADs (debits) and Direct Deposits (credits), each with distinct processing rules, timelines, and legal constraints.

2. Dual Submission Models: File transmission (batch-based via FTP) and PaymentStream AFT (real-time entry via UI)—both must converge on identical business logic.

3. Settlement Architecture: Separate handling of outgoing transactions (originated by member) and incoming transactions (received from other institutions), with posting rules tied to lead times and due dates.

4. Exception Lifecycle: Rejects (pre-posting validation failures), Recalls (pre-posting reversal requests), Returns (post-posting failures like NSF/invalid account), and NOCs (non-financial notifications of data corrections).

5. Service Charge Model: Transaction-level charges posted at month-end; tiered by institution class and currency (CAD/USD).

6. Regulatory Compliance: CPA rules (H1 for PADs), FINTRAC requirements, dispute/recourse procedures, and provincial variations (BC vs. Ontario reporting requirements).

How It Works

1. Enrollment & Setup: Financial institutions register for AFT service and configure policies (identification, record-keeping, member notifications).

2. File Creation & Submission: Originators create AFT files or use PaymentStream UI; files submitted before strict deadlines (varies by service type: typically 1-3 business days pre-due date).

3. Validation & Posting: Central 1 validates file format and transaction data; rejects defective files/transactions; posts valid outgoing transactions to originator's account and generates offsetting entries.

4. Settlement Processing: Incoming AFT transactions posted to receiving institution's Central 1 account by run, branch, and currency; up to 4 CAD runs daily.

5. Exception Handling: Failed transactions returned with reason codes; originators issue recalls (before posting) or error corrections (after posting); recipients submit NOCs for data updates.

6. Reporting & Reconciliation: MERG and Account Activity/Balance reports distributed via FTP; members reconcile via online channels.

Integration Points

  • File Exchange & Report Distribution: FTP service for file upload/download and report delivery
  • Settlement System: Central 1 GL posting engine for transaction and service charge settlement
  • Online Return System (ORS): Alternative return submission channel for non-file-transmission institutions
  • Batch Return Service (BC): Electronic return processing via in-house/third-party systems
  • PaymentStream Platform: Real-time AFT entry and validation interface
  • Legal/Compliance Framework: CPA rules, provincial legislation, dispute/recourse procedures

Architect Notes

  • Deadline Criticality: AFT deadlines are hard cutoffs tied to clearing cycles; no grace periods. Design submission validation with precise timestamp validation against Pacific Time zone.
  • Dual Posting Model: Outgoing transactions post twice (originator account + offsetting entry); incoming posts once. Settlement logic must account for both patterns without duplication.
  • Return Offset Options: Institutions can elect return-with-offset or return-no-offset; this affects GL posting counts and requires configurable settlement rules.
  • Legacy Contingency Paths: Paper-based and manual entry procedures exist for system outages; architecture must support graceful degradation to non-electronic workflows.
  • Provincial Variations: BC/Ontario reporting and service charge rules differ; ensure configuration flexibility for multi-jurisdictional members.
  • Dispute Windows: PAD recourse periods
