# Electronic Transactions: Solutions Architect Brief

Purpose

This section covers Central 1's transaction processing architecture for ATM, POS, and Interac e-Transfer channels. It defines settlement mechanics, fraud detection workflows, trace/adjustment procedures, and regulatory compliance requirements that underpin payment clearing and risk management across the credit union network.

Key Concepts

1. Network Topology: Three distinct transaction networks—THE EXCHANGE®, ACCULINK®, and Cooperative Node—each with different settlement rules, trace handling, and acquirer/issuer responsibilities. Network choice determines settlement timing and dispute resolution pathways.

2. Acquirer vs. Issuer Roles: Transaction handling diverges based on party role. ATM deposit procedures, trace initiation, and adjustment authority differ significantly between the institution owning the terminal (acquirer) and the one issuing the card (issuer).

3. Enterprise Fraud Management (EFM) System: Real-time transaction monitoring that generates alerts within incidents. Incidents feed operational workflows for fraud teams; alerts correlate to transaction IDs and trigger email notifications for investigation and disposition.

4. Settlement and Reconciliation: Daily net settlement posts to institutional Central 1 accounts with separate debit/credit totals. Currency conversion (USD→CAD) occurs at posting. Interac e-Transfer limits are configurable per Limit Group; settlement includes autodeposit tracking.

5. Trace and Adjustment Mechanics: Manual, direct peer-to-peer, or Central 1-mediated processes depending on network and transaction type. Response codes from banking host and Interac feed remediation workflows; timeout errors trigger Banking Host Remediation processes.

6. Regulatory Compliance: PCMLTFA/PCMLTF training and policies mandatory for ATM staff; card capture in dual custody; response code handling tied to AML/KYC obligations.

How It Works

1. Transaction initiated at ATM/POS or via Interac e-Transfer channel

2. EFM monitors transaction against fraud rules; generates alert if suspicious

3. If alert triggered: incident created, fraud team notified, analyst reviews and disposes

4. Transaction settles daily to institution account (net of all adjustments/traces)

5. If dispute or error: trace request initiated (peer-to-peer on THE EXCHANGE®, or Central 1-mediated on other networks)

6. Trace resolution updates settlement via adjustment posting

7. Reports generated daily (EMCU, EMCD, ECCX for e-Transfer; EFM incident/alert reports for fraud)

Integration Points

  • Banking Host: Remediation process for timeout errors; response code interpretation
  • Interac Systems: e-Transfer profile management (OAS), limit enforcement, autodeposit logic
  • Client Centre: Access control, fraud analyst roles, trace submission (OAS module)
  • Settlement Ledger: Daily posting of net amounts to institutional accounts
  • Card Networks: Interac, THE EXCHANGE®, ACCULINK® switching and clearing

Architect Notes

  • Dual Settlement Model: ATM deposits and POS transactions settle separately; USD conversion occurs post-transaction, not at posting—design reporting to handle this lag.
  • Trace Ownership: THE EXCHANGE® traces are peer-to-peer (no Central 1 intermediation); other networks require Central 1 involvement. Integration design must route correctly by network.
  • Remediation Complexity: Timeout-triggered retries can create duplicate transactions. Banking host must idempotently handle Central 1 resubmissions; this is a critical failure mode.
  • Fraud Incident State Machine: Incidents have open/closed states; filters and team restrictions apply. Design audit trail for all disposition changes.
  • Regulatory Audit Trail: Card capture, PCMLTFA training logs, and fraud decisions must be immutable and traceable—design for compliance inspection.
